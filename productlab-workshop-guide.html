Phase 2 · 1:30 – 2:00 PM · 30 minutes

Idea to Architecture

How to think before you build. Turning fuzzy concepts into a concrete, buildable spec — in under 15 minutes.

06
The OCP Framework: One Customer Problem
Proprietary Framework · Live Exercise · Joshua Theophilus

Before you open any tool, you complete the OCP. Twelve "ones" — each one forces a decision. No waffling, no hedging. One answer per box.

01
Customer Problem
The single most painful problem your target user faces today
02
Solution
Your product's specific answer — one sentence, no ands, no buts
03
Value Proposition
Why a user picks you over doing nothing — the tangible outcome you deliver
04
Killer Feature
The one capability that makes users try — and tell others — about your product
05
Monetisation Model
How this makes money — subscription, one-time, marketplace, or freemium
06
Core User Segment
The one type of person you build for first — be precise, not broad
07
Key Strategy
Your primary growth lever — virality, SEO, partnerships, community, or outbound
08
Key Channel
Where your users already live — LinkedIn, WhatsApp, X, communities, or email
09
Key Metric
The one number that tells you the product is working — activation, retention, revenue
10
Unfair Advantage
What you have that a competitor cannot easily copy — access, expertise, network, data
11
Retention Hook
Why users come back — habit, data lock-in, community, streaks, or ongoing value
12 · JT's Add
Growth Loop
How the product grows itself — referrals, SEO content, viral sharing, or network effects

The OCP flows from problem → solution → value → growth → retention in one pass. When all twelve align, you have something worth building. If they don't, you haven't found the right idea yet — and that's the point.

  • Live exercise: complete your OCP canvas in 10 minutes. Share 2–3 with the room for real-time critique.
  • Rule: if you can't answer all twelve in 10 minutes, the idea needs more thinking — not more building
  • Point participants to Product Nerve AI (productnerve.com) — they should validate their OCP output here before Lab A begins
All Roles
07
Prompt Engineering for Builders — The SPARK Framework
Core Skill Slide · Live Demo

The quality of what you build = the quality of your prompts. Introduce SPARK:

  • S – Scope: What are you building? (App? Landing page? Dashboard? API?)
  • P – Persona: Who is the user? What do they need to accomplish?
  • A – Actions: What are the 3 key things a user must be able to do?
  • R – Result: What happens when they complete each action?
  • K – Keep out: What should this NOT do? Scope creep kills MVPs.
  • Live demo: Prompt a live landing page in Lovable using SPARK. Watch it appear in real time.

All three labs run 90 minutes each, all facilitated by Joshua Theophilus. 10-minute breaks between each session.

Lab A · 2:40 – 4:10 PM
Lovable: AI Startup Directory
Build the live platform everyone lists their idea on — then submit yours.

The Concept

  • Build a live AI-Assisted Startup Directory using Lovable — a real, public-facing platform where participants submit and browse AI startup ideas built with AI-assisted tools
  • By the end of the lab, every participant submits their OCP idea from Phase 2 to the live directory
  • After the workshop, Joshua reviews all submissions and spotlights the best via Product Slice HQ
  • The directory stays permanently live — a community artefact from ProductLab Edition 01

What Gets Built

  • Homepage: browsable card grid of AI startup submissions with filters by category, tool used, and stage
  • Submit page: form capturing name, idea title, problem solved, tool used, live URL (if any), and one-liner OCP pitch
  • Supabase backend: all submissions stored in a real Postgres database, rendered in real time
  • Detail page: each idea gets its own public-facing page with the full OCP breakdown
  • Spotlight badge: "Featured by JT" tag applied post-workshop to the best submissions
  • Resend confirmation: every submitter gets a branded email confirming their idea is live
  • PostHog analytics: track page visits, most-viewed ideas, and submission velocity
  • Deployed to Vercel — live, shareable URL before the session ends

Step-by-Step (90 min)

  • 0:00–0:08 — Write and run the SPARK prompt in Lovable for the full directory. Show the initial output live.
  • 0:08–0:25 — Iterate: refine the card grid, fix submission form fields to match the OCP, tune the layout.
  • 0:25–0:40 — Connect Supabase: create ideas table (name, title, problem, tool, url, pitch, featured). Enable admin auth.
  • 0:40–0:50 — Wire Resend: submission confirmation email auto-fires to every founder who lists their idea.
  • 0:50–1:00 — Drop PostHog snippet in. Deploy to Vercel. Test full submission flow end-to-end.
  • 1:00–1:20 — Open submissions: every participant fills in and submits their OCP live. Watch the directory populate in real time.
  • 1:20–1:30 — Browse together. JT announces which ideas will receive the post-workshop spotlight. Prep 2-min demo.

Why This Format Works

  • Participants don't just watch a build — they become part of the product they just watched being built
  • The directory is proof of concept AND the community artefact from Edition 01
  • The spotlight mechanic gives every participant a reason to stay engaged with Product Slice HQ after the workshop
  • JT owns a growing, curated directory of AI startup ideas to feature, write about, and build community around

Lab B · 4:20 – 5:50 PM
Google AI Studio + Stitch + Firebase
Design-first, AI-powered — from visual intent to a Gemini-driven product.

The Concept

  • Build "Smart Feedback Collector" — a product feedback tool where users submit feedback, Gemini AI categorises it by theme and sentiment, and results render live in a visual dashboard
  • Three Google tools working together: Stitch for design → AI Studio + Gemini for intelligence → Firebase for backend data storage
  • Google Antigravity introduced as an optional adjacent tool for further exploration after the workshop

What Gets Built

  • Submission screen: clean form UI built and exported from Stitch — real HTML/CSS/React, not a mockup
  • Gemini AI layer: feedback text is sent to Gemini via AI Studio API → returns category tag + sentiment score (positive / neutral / negative)
  • Firebase Firestore: all submissions + AI analysis results stored in real time, no server required
  • Dashboard screen: live feed of categorised feedback with sentiment visualisation — built in Stitch, powered by Firebase
  • Optional: Google Antigravity introduced briefly as an alternate build path for AI-native apps

Step-by-Step (90 min)

  • 0:00–0:10 — Define the 3 core screens: Submit Feedback, Thank You, Dashboard. Brief the Stitch prompt.
  • 0:10–0:30 — Use Stitch to generate all three screens. Export the React/HTML code. Review structure live.
  • 0:30–0:50 — Open Google AI Studio: create a Gemini prompt that takes feedback text and returns a JSON object with category + sentiment. Test with live examples.
  • 0:50–1:05 — Wire the Gemini API call into the submission form: on submit → call Gemini → store result in Firebase Firestore.
  • 1:05–1:20 — Connect dashboard to Firebase: real-time listener renders new submissions as they arrive. Show live data flow.
  • 1:20–1:25 — Quick Antigravity mention: show what it does and where it fits as an alternative build path.
  • 1:25–1:30 — Test full flow. Prep 2-min demo.

Key Talking Points

  • "Stitch is what happens when Figma and a frontend dev have a child — powered by Gemini"
  • "Everything you see is real code. Not a wireframe. You can deploy this today."
  • "Firebase gives you a real-time backend without writing a single server function"
  • "This trio — Stitch + AI Studio + Firebase — is a full AI-native product stack from Google, end to end"

Lab C · 6:00 – 7:30 PM
Claude Code + Adalo
Two builds in one lab — a web app via Claude Code Desktop, and a mobile app via Adalo.

The Concept

  • Lab C runs two sequential live builds, splitting the 90 minutes between Claude Code (web) and Adalo (mobile)
  • Claude Code build (50 min): GTM Campaign Planner — a structured web app that helps teams build, organise, and output a shareable go-to-market campaign brief
  • Adalo build (35 min): Booking & Appointment App — a native mobile app with embeddable booking widget that can be integrated into any website or platform via a simple code snippet
  • All done via the Claude Desktop App — no terminal setup, no VS Code required

Build 1 — GTM Campaign Planner (Claude Code · 50 min)

  • Users input product name, target audience, key message, and campaign goal
  • App structures and outputs a complete GTM brief: channel plan, messaging hierarchy, CTA recommendations, and success metrics — formatted as a shareable, downloadable doc
  • Supabase stores all campaigns. Resend emails the finished brief to the user.
  • Deployed to Vercel — live URL in under 50 minutes via Claude Desktop prompting only

Build 1 — Step-by-Step (50 min)

  • 0:00–0:08 — Open Claude Desktop App. OCP for the GTM Planner. Prompt Claude to scaffold the full project.
  • 0:08–0:20 — Review file structure. Prompt auth: "Add Supabase email login and a protected dashboard."
  • 0:20–0:35 — Prompt the core form: "Build a GTM brief form with fields for product name, audience, key message, channels, and goal. Save to Supabase."
  • 0:35–0:45 — Prompt the output page: "Generate a structured GTM doc from the form data. Make it downloadable as PDF. Email it via Resend."
  • 0:45–0:50 — Push to GitHub. Deploy to Vercel. Show live URL. Test the full flow end-to-end.

Build 2 — Booking & Appointment App (Adalo · 35 min)

  • Native mobile app for iOS & Android — service listing, date/time picker, booking confirmation, and appointment history
  • Built entirely in Adalo's visual editor — no code written
  • Embeddable booking widget: Adalo generates a web embed snippet that can be dropped into any website, Notion page, or external platform
  • Use case: any service business (consultant, coach, clinic, studio) can embed this and take bookings without a custom dev build

Build 2 — Step-by-Step (35 min)

  • 0:50–1:00 — Create Adalo project. Set up collections: Services, Users, Bookings. Define relationships.
  • 1:00–1:15 — Build core screens: Home (service list), Book (date/time picker + confirm), My Bookings (history). Link all screens.
  • 1:15–1:20 — Enable Adalo auth. Test login + booking flow on mobile preview.
  • 1:20–1:25 — Show the web embed snippet. Paste into a live webpage and demonstrate the booking flow embedded in a browser.
  • 1:25–1:30 — Recap both builds. Close with: "One lab, two live products, zero lines of manually written code."

Key Talking Points

  • "Claude Code via Desktop App is the most accessible entry point to agentic coding — no setup friction"
  • "You directed. Claude executed. Every prompt you wrote was a spec — and specs became working software."
  • "Adalo's embed feature means your mobile app is also a widget. One build, multiple surfaces."
  • "By the end of this lab you've shipped a web app and a mobile app in 90 minutes. That is the new baseline."
